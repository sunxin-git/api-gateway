下面是我重新按 GitHub 项目、官方文档、社区/第三方资料交叉筛过后的结论。**按你这个场景，我会优先选 New API 做主干；如果高并发和性能优先，则重点看 Bifrost；如果未来要大量接海外模型商和 Vertex/Veo/OpenAI 等，则 LiteLLM 很适合作为适配层或参考实现。**

## 一句话结论

**最贴近你需求的是：New API > Bifrost ≈ LiteLLM > Portkey > One API > Kong AI Gateway。**

原因是你的核心不是传统 LLM 聚合，而是“多媒体模型商业化网关”：视频/图片任务、异步任务生命周期、模型特有参数透传、积分/倍率/加价计费、渠道并发、日志与图形分析。New API 已经有中文生态、分销/额度/计费/仪表盘/渠道管理底座，并且文档里明确支持 image/audio/video 接口、Midjourney/Suno/视频模型、OpenAI/Claude/Gemini 转换、自定义上游等能力；它还继承/兼容 One API 数据体系，所以比 One API 更适合作为二开起点。

## 1. New API：我认为最适合做主干

New API 的定位非常接近你要的东西：它是一个 AI API 网关和用量管理系统，支持 OpenAI、Anthropic、Google Gemini、DeepSeek、Midjourney、Suno 等多种服务，并提供统一接口、智能路由、负载均衡、成本管理、安全、数据仪表盘、多租户能力。

它的 GitHub README 里明确列出：数据仪表盘、权限管理、用户/令牌分组、模型限制、充值/配额、Stripe/易支付、组织级请求或用量计费、缓存计费、灵活计费策略等能力。 它也支持 OpenAI Responses、Realtime、Claude Messages、Gemini、Rerank 等格式，以及加权随机、自动重试、用户级模型限流、OpenAI/Claude/Gemini 格式转换。

对你这个“多媒体为主”的场景，New API 的一个关键优点是：它已经把接口范围扩展到 chat、responses、image、audio、video、embeddings、rerank、realtime、Claude、Gemini 等，并支持 OpenAI 兼容接口、Midjourney Proxy、Suno、自定义上游等。 文档里还列出视频生成相关支持，例如 Sora、阿里 Wanxiang、Kling、即梦、Vidu 等，并支持 text-to-video、image-to-video、keyframe-to-video、audio combining，以及 watermark、prompt_extend、audio、seed 等视频特有参数。

**但它还不是“完全现成”。** 你需要二开的部分主要是：

1. **模型能力 Schema 注册表**：每个模型配置自己的参数、枚举、默认值、可见性、校验规则。比如 Seedance 2.0 支持 text、image、audio、video 输入，且有音视频联合生成和参考/编辑能力；这类模型不能简单用 LLM 的 `messages` 或普通 image API 覆盖。
2. **视频异步任务生命周期**：`create_task / query_task / cancel / callback / download / retry / expire`。
3. **商业化计费引擎**：New API 已有额度/成本/计费底座，但“某功能 10 积分 + 2 积分 = 12 积分”这种固定加价策略，最好抽象成独立 PricingPolicy。
4. **并发池/排队系统**：New API 有模型限流、加权路由、重试，但“给某个模型/渠道/租户分配 N 个并发任务”尤其是视频长任务，还需要额外 semaphore/queue。
5. **模型调用日志分析增强**：New API 有仪表盘，但如果你要按模型、任务类型、分辨率、时长、失败原因、租户、渠道、利润率做图形化分析，建议接 ClickHouse/Prometheus/Grafana 或自建 BI 表。

**结论：** 如果你们优先做商业化平台、积分、账单、渠道管理、客户管理，我会把 New API 作为第一候选。

------

## 2. Bifrost：性能、高并发、二开友好度最强的候选

Bifrost 是 Go 写的 AI Gateway，强调高性能、OpenAI-compatible 接口、多 Provider、fallback、load balancing、semantic caching、实时监控和 analytics。官方 README 里列出了多 Provider、自动 fallback、负载均衡、MCP、语义缓存、多模态、自定义插件、usage tracking、rate limiting、预算管理、虚拟 Key、Prometheus/OpenTelemetry 等能力。

它的性能宣传很激进：文档里声称 sustained 5000 RPS、11µs overhead；官网还声称 5000 req/s、20µs added latency、超过 1000+ teams 使用。这个属于厂商自述，需要你们压测验证，但从架构和语言选择看，它确实比 Python 类网关更适合做高并发热路径。

Bifrost 对你有一个很重要的点：它的 release notes 提到过 OpenAI image editing/variations 支持，并修复/增强了 custom provider 的参数过滤和 `ExtraParams` 透传。这和你说的“业务系统要能暴露不同模型的特有参数”非常相关。

**适合的架构：**

如果你们视频任务量很大、并发分配很重要，可以考虑：

**业务系统 → 自研 Media API 层 → Bifrost/Go 路由层 → 各模型 Provider**

或者：

**New API 负责商业化/用户/账单/管理后台，Bifrost 负责高并发路由和通道层。**

**短板：** Bifrost 的商业化账单、充值、积分、客户后台，不如 New API 直接；模型覆盖和多媒体模型现成适配也不如 LiteLLM/New API。它更像“高性能 AI 网关内核”，不是完整分销平台。

------

## 3. LiteLLM：模型适配和国际 Provider 覆盖最强

LiteLLM 是当前最活跃、覆盖面最广的开源 AI Gateway 之一。GitHub README 显示它提供统一 OpenAI 格式接口，覆盖 100+ LLM providers，支持 SDK 和 Proxy，提供 virtual keys、spend tracking、guardrails、load balancing、admin dashboard，并声称 1k RPS 下 P95 约 8ms。 它支持 `/chat/completions`、`/responses`、`/embeddings`、`/images`、`/audio`、`/batches`、`/rerank`、`/messages` 等端点。

LiteLLM 的预算、虚拟 Key、花费追踪比较成熟：文档中有 personal/team/team member/agent budgets、global proxy max_budget、virtual key spend、user/team/project spend 等说明。 路由策略也比较丰富，包括 simple-shuffle、least-busy、usage-based-routing、latency-based-routing、retry、cooldown 等。

对你未来接 Google Veo 路线也有价值：LiteLLM 文档已经有 Vertex AI Veo 视频生成支持，并说明其 OpenAI-style video API、duration-based pricing、request/response logging、virtual keys、budget tracking、rate limiting 等能力。 另外，截至这次检索，我没有看到 Google 官方 Veo 4 发布资料；Google 官方开发者资料显示的是 Veo 3、Veo 3 Fast 稳定用于 Gemini API，以及 Veo 3.1/3.1 Fast 在 Gemini API paid preview。

**但 LiteLLM 不太适合直接做商业化积分平台。** 它更适合作为“模型 Provider 适配层”，而不是最终的“积分、充值、客户账单、利润分析、渠道并发分配”平台。并且我检索到过与自定义图像/视频定价相关的 issue，说明图片/视频的自定义成本核算要在你们上线前重点验证。

**结论：** LiteLLM 很适合做参考实现或作为部分 Provider 的 adapter，尤其是 OpenAI、Gemini/Vertex、Anthropic、Bedrock 这类海外模型；但主平台我不建议只靠 LiteLLM。

------

## 4. Portkey Gateway：企业级能力强，但不一定适合纯开源二开

Portkey Gateway 2.0 的 README 提到正在把企业网关能力合并进开源项目，并提供覆盖 2300+ models、40+ providers 的模型价格信息。 它声称支持 1600+ language、vision、audio、image models，提供 retries、fallbacks、load balancing、conditional routing、guardrails、multimodal、MCP 等能力。

Portkey 的优势在于“企业级 AI 网关”能力：usage analytics、request volume、latency、cost、error rates、routing、guardrails、安全 Key 管理、RBAC、缓存等都比较完整。 它的 pricing schema 对多媒体也比较友好，支持 web_search、file_search、thinking_token、image_token、不同分辨率/质量的 image pricing、video_duration_seconds 等复杂计价单元。

**主要问题：** Portkey 文档里 API key budget/rate limit 标注为 Enterprise 和部分 Pro 用户能力，不一定是你拿开源仓库就能完整自托管的能力。 如果你们愿意采购/深度绑定 Portkey，它很强；如果目标是“完全开源可控 + 大量二开”，我会把它排在 New API、Bifrost、LiteLLM 后面。

------

## 5. One API：可以参考，但不建议作为新项目主干

One API 是老牌中文模型 API 管理项目，支持多模型、负载均衡、流式、令牌管理、兑换码、渠道管理、用户/渠道分组、模型列表配置、额度明细等。 它的计费公式很清晰：额度消耗 = 分组倍率 × 模型倍率 × token 数量，并支持 completion token 单独倍率。

这正好对应你说的“倍率方式加价”。但 One API 的更新活跃度、多媒体接口、图形化分析、未来扩展性都不如 New API。New API 本身也说明与 One API 数据库兼容，所以除非你们非常看重 MIT 协议，否则更建议直接从 New API 起步。

------

## 6. Kong AI Gateway：如果你们已有 Kong 体系才值得重点考虑

Kong AI Gateway 适合已有 Kong/Konnect/API 管理体系的团队。它提供 routing、load balancing、health checks、认证、限流、转换、日志监控，以及 OpenAI、Anthropic、Gemini、AWS Bedrock、Azure、Mistral、Hugging Face 等 AI provider 的路由能力。 Kong 文档中也列出 Universal API、AI rate limiting、semantic caching、semantic routing、MCP、guardrails、LLM metrics、metering & billing、cost control 等 AI 网关能力。

但它的问题也明显：Kong 的 AI Rate Limiting Advanced 等能力需要 AI License，并且它更偏通用 API 网关插件体系，而不是多媒体模型商业化平台。 你们如果没有 Kong 基础设施，我不建议把它作为主项目。

## 我建议的技术路线

### 推荐路线 A：New API 主干 + 自研多媒体模型适配层

这是我认为最贴近你们需求的方案。

架构可以是：

```
业务系统
  ↓
统一 Media Model API
  ↓
New API 二开版：用户 / Key / 权限 / 额度 / 账单 / 日志 / 仪表盘
  ↓
模型能力 Schema Registry
  ↓
Provider Adapter 层
  ├─ ByteDance Seedance 2.0
  ├─ ByteDance Seedream
  ├─ Kling / 可灵
  ├─ OpenAI GPT Image 2
  ├─ Google Vertex / Veo
  ├─ 第三方平台 A
  └─ 第三方平台 B
```

OpenAI 官方文档显示 GPT Image 2 是图像生成和编辑模型，输入支持 text/image，输出 image，支持灵活图片尺寸和高保真图像输入；这类模型和 Seedance 2.0 这种 text/image/audio/video 输入的视频模型，参数结构完全不同，所以必须做“模型能力 Schema”，不能只做统一 OpenAI-compatible 转发。

### 推荐路线 B：New API 做商业化后台，Bifrost 做高并发通道层

如果你们视频任务非常多，且每个模型/渠道都有严格并发限制，可以考虑：

```
业务系统
  ↓
New API 二开版：用户、账单、额度、价格、日志
  ↓
任务队列 / 并发调度器
  ↓
Bifrost：高性能 Provider 路由、fallback、观测
  ↓
模型供应商
```

这条路线的好处是：New API 解决商业化和中文后台，Bifrost 解决高并发和路由性能。坏处是系统复杂度更高，需要你们团队有较强后端工程能力。

### 推荐路线 C：LiteLLM 作为 Provider Adapter 参考或子模块

LiteLLM 可以不作为主平台，但非常适合在接海外模型时参考或复用。例如 OpenAI、Gemini、Vertex、Anthropic、Bedrock 等，LiteLLM 的 provider mapping、virtual key、budget、routing、logging 设计都值得借鉴。尤其是未来接 Veo 相关模型时，LiteLLM 的 Vertex/Veo 支持可以作为参考。

## 你们二开时最关键的 6 个模块

### 1. 模型能力 Schema Registry

每个模型不要只配置 `model_name` 和 `price`，而是要配置完整能力：

```
{
  "model": "seedance-2.0",
  "provider": "bytedance",
  "tasks": ["text_to_video", "image_to_video", "audio_video_generation", "video_edit"],
  "input_modalities": ["text", "image", "audio", "video"],
  "output_modalities": ["video"],
  "params_schema": {
    "duration": { "type": "integer", "enum": [5, 10] },
    "resolution": { "type": "string", "enum": ["720p", "1080p"] },
    "aspect_ratio": { "type": "string", "enum": ["16:9", "9:16", "1:1"] },
    "seed": { "type": "integer" },
    "watermark": { "type": "boolean" },
    "reference_images": { "type": "array" },
    "reference_audio": { "type": "file" },
    "reference_video": { "type": "file" }
  }
}
```

这里的字段只是示例，实际要按官方 API 或第三方平台 API 落地。重点是：**业务系统应该能读取这个 Schema，知道某个模型到底支持哪些特殊参数。**

### 2. 参数透传与参数转换

不同模型参数名可能不同：

```
业务参数：aspect_ratio
Provider A：aspectRatio
Provider B：ratio
Provider C：size = "1920x1080"
```

所以适配器要做三件事：

```
validate → transform → provider_request
```

不要让业务系统直接硬编码每个供应商参数，否则后面换模型会很痛苦。

### 3. 多媒体异步任务系统

视频模型通常不是同步 HTTP 返回结果，而是异步任务：

```
create_task → polling/status → success/failed → download artifact → callback
```

建议统一暴露：

```
POST /v1/media/tasks
GET  /v1/media/tasks/{task_id}
POST /v1/media/tasks/{task_id}/cancel
GET  /v1/media/tasks/{task_id}/artifacts
```

内部记录：

```
internal_task_id
upstream_task_id
provider
model
tenant_id
status
params_snapshot
pricing_snapshot
input_files
output_files
failure_reason
created_at
completed_at
```

### 4. 积分与商业计费策略

你提到两类计费：倍率加价、固定积分加价。建议设计成统一公式：

```
charge_credits =
  round(
    base_credits * multiplier
    + fixed_markup_credits
    + feature_markup_credits
    + tenant_markup_credits
  )
```

例子：

```
基础功能：10 积分
固定加价：2 积分
最终扣费：12 积分
```

视频/图像还要支持不同单位：

```
per_request
per_image
per_video_second
per_video_task
per_resolution
per_duration
per_input_token
per_output_token
per_image_token
provider_cost_usd
```

Portkey 的 pricing schema 已经支持 image token、video duration seconds、不同图像质量/分辨率等复杂单元，可以作为你们计费模型设计参考。 One API/New API 的倍率/额度体系也可以作为中文积分体系的基础参考。

### 5. 并发池与排队

这块现有开源项目大多只有 rate limit，不一定有你要的“模型并发数分配”。

建议设计：

```
concurrency_bucket:
  model_id
  provider_id
  channel_id
  tenant_id
  max_inflight
  current_inflight
  queue_size
  priority
  timeout_seconds
```

例如：

```
Seedance 2.0 官方账号 A：最多 10 个并发任务
Seedance 2.0 第三方渠道 B：最多 5 个并发任务
租户 X：最多 3 个并发任务
租户 Y：最多 1 个并发任务
```

视频任务要注意：**并发占用不等于 HTTP 请求占用**。任务提交成功后，直到上游任务完成前，都应该占用一个模型任务并发位。

### 6. 日志、账单、图形化分析

至少要记录这些维度：

```
tenant_id
user_id
api_key_id
model_id
provider_id
channel_id
task_type
request_params_snapshot
provider_raw_usage
normalized_usage
base_cost
charged_credits
profit
latency
queue_wait_time
status
failure_code
failure_reason
created_at
completed_at
```

图形化分析建议按这些看板做：

```
模型调用量趋势
模型成功率 / 失败率
平均生成时长
队列等待时间
各渠道成本
各模型利润
租户消费排行
失败原因排行
分辨率 / 时长 / 功能分布
并发池占用率
```

New API 自带数据仪表盘和统计分析底座，可以先用它起步；后续如果数据量大，再接 ClickHouse + Grafana/Metabase。

## 最终建议

**如果你们近期就要落地，我建议：**

1. **主项目选 New API 二开。**
    它最贴近“商业化、多租户、额度、账单、渠道、仪表盘、中文生态”的需求。
2. **为多媒体模型新增独立 Media Adapter 模块。**
    不要强行塞进传统 chat/completions 逻辑里。
3. **新增模型能力 Schema Registry。**
    让业务系统能动态知道 Seedance、Seedream、GPT Image 2、Kling、Veo 等模型各自支持什么参数。
4. **新增 PricingPolicy。**
    同时支持倍率、固定积分加价、按秒、按张、按分辨率、按功能加价。
5. **新增视频任务队列与并发池。**
    这是多媒体模型商业化网关的核心，不只是普通 API rate limit。
6. **高并发场景引入 Bifrost 或自研 Go/Rust 调度层。**
    如果只是中等规模，New API 二开就够；如果视频任务量大，Bifrost 更适合做通道层。

**我的最终排序：**

```
首选主干：New API
性能增强/高并发通道：Bifrost
海外模型适配参考/子模块：LiteLLM
企业级备选：Portkey
历史参考/轻量备选：One API
已有 Kong 体系才考虑：Kong AI Gateway
```

如果只能选一个项目开始，我会选 **New API**。
 如果你们团队后端能力强、并发压力大，我会选 **New API + Bifrost/自研调度层**。
